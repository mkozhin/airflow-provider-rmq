# Четыре цикла потребления сообщений — не унифицировать

В кодовой базе четыре реализации паттерна «получить сообщение → проверить фильтр → ACK/NACK»:
- `hooks/rmq.py` — blocking pika, возвращает список сообщений
- `sensors/rmq.py` — poke-цикл, NACK не совпавших (намеренно), возвращает `bool`
- `triggers/rmq.py` — async aio_pika, polling, возвращает `TriggerEvent`
- `watcher/consumer.py` — async aio_pika, push-итератор, запускает DAG

Это не дублирование — это четыре семантически разных операции с разными стратегиями ACK и
разными моделями выполнения (sync blocking vs async).

NACK не совпавших сообщений в sensor — намеренное поведение для сценариев с несколькими
типами сообщений в одной очереди (разные consumers берут своё).

## Considered Options

Рассматривалась унификация через общий `MessageConsumer` протокол. Отклонено:
sync (pika) и async (aio_pika) — фундаментально разные модели выполнения,
общая абстракция поверх обоих была бы сложнее, чем четыре явных реализации.
Боли от дублирования нет, риск сломать обратную совместимость — реальный.
