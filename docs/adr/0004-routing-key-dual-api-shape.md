# `routing_keys` и `routing_key_ids`×`routing_key_status` сосуществуют, не заменяют друг друга

`@rmq_trigger(exchange=...)` принимает routing key двумя способами: `routing_keys: list[str]`
(литеральные topic-ключи любой формы) и `routing_key_ids` × `routing_key_status` (Jetstat-форма
`{id}.{status}`, по умолчанию `status="*"`). Оба можно указывать одновременно — итоговый набор
ключей — их объединение.

Причина: исходная формулировка задачи ("routing keys полностью задаются декоратором, без
привязки к Jetstat") не совпадала с фактическим API — `routing_key_ids`×`status` жёстко
отражает двух-сегментную семантику Jetstat (id задачи + статус) и не описывает другие формы
routing key (три сегмента, без понятия "status" вообще). Единственный реальный потребитель
сегодня — Jetstat, но фиксировать domain-agnostic декоратор на одной domain-specific форме —
сделать API недорого расширяемым нельзя постфактум без breaking change (это публичная
сигнатура).

## Considered Options

- Оставить только `routing_key_ids`×`status` (как в первой версии плана). Отклонено: бесплатно
  закрывает дверь для будущих потребителей с другой формой ключа.
- Заменить `routing_key_ids`×`status` на единственный `routing_keys: list[str]`. Отклонено:
  убирает удобный cross-product синтаксис для текущего (и единственного) реального случая.
- Оставить оба способа, объединять результат (выбрано). Чуть больше валидации/тестов
  взамен на отсутствие компромисса между гибкостью и удобством для Jetstat-случая.
